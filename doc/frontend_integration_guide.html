fplaces Frontend Integration Guide

Backend Engineer: Sunday Ajayi

This guide breaks down the division of responsibilities between the stateless REST API (used for fetching static data and initiating actions) and the WebSocket layer (used exclusively for receiving real-time live updates).

All REST endpoints require the Authorization: Bearer <access_token> header, and all WebSockets require the token to be passed as a query parameter ?token=<access_token>.

PART 1: Fan / User Application

This section covers the standard mobile or web application used by venue visitors.

1. Setup, Authentication, & Onboarding (REST)

The authentication flow uses SimpleJWT. The backend is stateless, so your client must store the access and refresh tokens securely.

  • Register: POST /api/users/register/ (Provide email and password). User is created but is_email_verified is False.
  • Verify Email: POST /api/users/verify-email/ (Provide email and the 6-digit otp sent to their email).
  • Profile Setup: PATCH /api/users/me/ (Provide pseudo_name). Required before the user can post.
  • Standard Login: POST /api/users/login/ (Provide email and password). Returns JWT tokens.
  • Google Login: POST /api/users/login/google/ (Provide Google OAuth id_token). Automatically registers/logs in the user and returns the standard JWT tokens.

2. Venue Selection & Initial Feed Load (REST)

When a user opens the app and selects a venue, fetch the historical state of that venue.

  • List Venues: GET /api/forum/venues/ (Fetch available stadiums/arenas).
  • List Sections/Categories: GET /api/forum/sections/?venue=<venue_id> and GET /api/forum/categories/.
  • Fetch Initial Feed: GET /api/forum/posts/?venue=<venue_id> (Paginated historical posts).

3. The Live Feed & Heatmap (WebSocket)

Immediately after fetching the initial feed, establish a WebSocket connection to the venue's room:
ws://<domain>/ws/venues/<venue_id>/?token=<access_token>

You never send data up this socket. It is purely for receiving live JSON broadcasts:

  1. new_post:
    • Payload: The full JSON object of the newly created post.
    • Action: Unshift/prepend it to your feed state.
  2. upvote_update:
    • Payload: {"post_id": 123, "upvotes_count": 5, "upvoted": true}
    • Action: Find post 123 in your state and update its count.
  3. post_hidden:
    • Payload: {"post_id": 123}
    • Action: Immediately remove post 123 from the UI.
  4. section_heat_update:
    • Payload: {"heatmap": [{"section_id": 4, "post_count": 8, "priority": "high"}, ...]}
    • Action: Overwrite your entire UI venue map state with this array. Map the priority string ("high", "medium", "low") directly to your CSS glowing colors. Any missing section ID has 0 posts and is neutral.

4. User Interactions (REST)

When the user does something, fire a REST request. If successful, the backend automatically broadcasts the result over the WebSocket.

  • Create Post: POST /api/forum/posts/ (Provide content max 140 chars, venue, category, and optional section). Triggers new_post and section_heat_update on the socket.
  • Toggle Upvote: POST /api/forum/posts/<post_id>/upvote/. This is idempotent. Triggers upvote_update on the socket.
  • Flag Post: POST /api/forum/posts/<post_id>/flag/ (Provide optional reason). If it hits 3 unique flags, it auto-hides and Triggers post_hidden and section_heat_update on the socket.

5. Personal Notifications (WebSocket & REST)

Users have a personal WebSocket for cross-venue, account-level alerts.

  • Connect: ws://<domain>/ws/notifications/?token=<access_token>
  • WebSocket Event (new_notification): Pushes a notification object in real-time so you can increment the unread badge count.
  • REST Fetch: GET /api/notifications/ (List paginated notification history).
  • REST Mark Read: POST /api/notifications/<id>/mark-read/ or POST /api/notifications/mark-all-read/.

PART 2: Admin / Moderator Dashboard

This section covers the web-based administrative panel used by staff. Accessing these endpoints requires an account with is_staff=True or user_type=admin.

1. Dashboard Authentication (REST)

Admins log in using the exact same login endpoints as users (POST /api/users/login/). Ensure your dashboard checks if the decoded JWT belongs to an admin before allowing entry.

2. Platform Statistics Overview (REST)

Used to render the charts and metric cards on the admin homepage.

  • Fetch Dashboard Stats: GET /api/admin/stats/
    • Returns: Aggregated metrics including total registrations, verified users count, total posts/comments, and categorical breakdowns of activity.

3. Content Moderation (REST)

Admins have elevated visibility into all content, including posts hidden from the public.

  • View Moderation Queue: GET /api/admin/posts/flagged/
    • Fetches posts sorted descending by flags_count. This is your primary moderation queue.
  • Hide Post: POST /api/admin/posts/<post_id>/hide/
    • Removes the post from the public feed immediately. Triggers the post_hidden and section_heat_update WebSocket events for active fans.
  • Show/Unhide Post: POST /api/admin/posts/<post_id>/show/
    • Reverses a manual or auto-hide.
  • Clear Flags: POST /api/admin/posts/<post_id>/clear-flags/
    • Resets a post's flags_count to 0 and archives the flag tickets after a moderator has reviewed it and deemed it safe.

4. User Management (REST)

Complete control over user accounts.

  • List Users: GET /api/admin/users/ (Fetch all users, paginated).
  • Archive User: DELETE /api/admin/users/<user_id>/ (Soft-deletes the user, disabling their account).
  • Restore User: POST /api/admin/users/<user_id>/restore/
  • Promote to Staff: Use standard PUT or PATCH on /api/admin/users/<user_id>/ to set user_type="admin" or is_staff=True.
